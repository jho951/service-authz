# CLAUDE.md

이 저장소는 **대학교 수강신청 시스템(REST API 서버)** 과제 구현을 위한 레포입니다.  
AI 에이전트(Codex/ChatGPT 등)는 아래 규칙을 반드시 지킵니다.

---

## 0) 최우선 목표(우선순위)

1. **항상 실행 가능한 상태 유지**
    - 서버가 정상 구동되고, `GET /health`가 **항상 200 OK**를 반환해야 한다.
2. **핵심 기능 우선 구현**
    - 수강신청/취소, 정원 초과 방지(동시성), 학점 제한(18), 시간 충돌 방지.
3. **사고의 깊이는 문서로 증명**
    - 요구사항 해석과 결정 사항은 `docs/REQUIREMENTS.md`에 기록한다.
    - API 명세는 `docs/API.md`에 “평가자가 문서만으로 테스트 작성 가능” 수준으로 작성한다.
4. **검증 가능한 결과물**
    - 동시성 테스트(정원 1, 동시 100 요청 → 성공 1)를 반드시 포함한다.

---

## 1) 기술/구현 제약

- Language: **Java**
- Framework: **Spring Boot**
- Build: **Gradle**
- API: **REST**
- Storage: **H2 in-memory + JPA** 또는 **인메모리**
    - 선택 근거와 한계/확장 방향은 `docs/REQUIREMENTS.md`에 명시한다.
- 데이터 생성: 서버 시작 시 **1분 이내**
    - 학과 10+, 강좌 500+, 학생 10,000+, 교수 100+

---

## 2) 필수 산출물 및 위치

- `README.md` : 빌드/실행/테스트 방법, 포트, `/health` 안내
- `docs/REQUIREMENTS.md` : 요구사항 분석, 불명확한 부분에 대한 결정(가정), 동시성 전략
- `docs/API.md` : 엔드포인트별 요청/응답/에러(상태코드+에러코드+예시)
- `prompts/` : AI 입력 프롬프트 이력(누락 금지)
- `src/` : 소스 코드 및 테스트 코드

---

## 3) 작업 방식(필수 프로세스)

AI는 아래 순서를 지켜 작업한다.

### Step A. 먼저 “설계”를 제시
- 요구사항 요약
- 불명확한 부분 목록 + 합리적 결정(가정)
- 데이터 모델(학생/강좌/교수/수강신청)
- 동시성 제어 전략(대안 비교 → 최종 선택)
- 규칙 우선순위(정원/학점/시간충돌/중복신청/취소)
- 시간/공간 복잡도(핵심 로직)

### Step B. 테스트 계획 제시 후 구현
- 동시성 테스트(정원 1 + 100 동시 요청) 우선 작성 계획
- 학점/시간충돌/중복신청 테스트 최소 1개씩

### Step C. 구현은 “최소 변경(diff)” 원칙
- 한번에 큰 리팩터링 금지
- 작은 단위로 안전하게 변경
- 빌드/테스트 깨지지 않게 유지

---

## 4) 동시성/정합성 규칙(필수)

### 반드시 지켜야 하는 핵심
- 정원이 1 남은 강좌에 100명이 동시에 신청해도 **정확히 1명만 성공**
- 정원 초과는 절대 불가

### 동시성 범위(기본 가정)
- 동일 강좌에 대한 동시 신청은 안전해야 한다.
- 동일 학생의 동시 신청(서로 다른 강좌 포함)도 정합성이 유지되어야 한다.
- 교착을 피하기 위한 락 획득 순서가 필요하면 문서에 기록한다.
