# REQUIREMENTS — 요구사항 분석 및 결정 사항 (Stable Spec)

본 문서는 기획팀 요청사항이 불완전한 상태이므로,
질문 없이 구현/테스트가 가능하도록 요구사항을 **안정 규격(Stable Spec)** 으로 고정한다.
원문과 충돌하지 않는 범위에서 합리적 결정을 내렸고, 모든 결정은 근거와 함께 기록한다.

---

## 1. 목표 및 범위

### 1.1 목표
- 수강신청 기간의 동시 요청 상황에서도 **정원(capacity)을 절대 초과하지 않는** 수강신청 REST API 제공
- 평가자가 문서만 보고 테스트를 작성할 수 있도록 **API/에러/정책**을 명확히 정의

### 1.2 구현 우선순위(평가 기준 반영)
1) **동작 여부**: 빌드/실행 + `GET /health` 200 + 기본 API 동작
2) **핵심 기능**: 정원/학점/시간충돌/중복 + **동시성 제어 검증**
3) **사고의 깊이**: AI 활용 기록(prompts) + 설계 문서 + 테스트 + 커밋 분리

### 1.3 구현 범위(MVP)
- 학생 목록 조회
- 강좌 목록 조회(전체/학과별, 정원/현재 인원/시간/학점 포함)
- 교수 목록 조회
- 수강신청
- 수강취소
- 내 시간표 조회(이번 학기)
- 헬스체크: `GET /health` (항상 200 OK)

### 1.4 비범위(문서로만 남김)
- 인증/인가(로그인, 권한)
- 선수과목, 학년 제한, 취소 기한 등 추가 정책
- 대기자/대기열(웨이팅)
- 팀티칭(강좌당 다교수)

---

## 2. 불변 규칙(Invariants)

아래 규칙은 어떤 상황에서도 항상 참이어야 한다.

1) **정원 초과 금지**
- 어떤 동시 요청 상황에서도 `enrolled <= capacity`
- 정원이 1 남은 강좌에 100명이 동시에 신청해도 **성공은 정확히 1건**이어야 한다.

2) **학점 제한**
- 학생의 이번 학기 신청 학점 총합은 `<= 18`

3) **시간표 충돌 금지**
- 동일 학생이 신청한 두 강좌의 시간이 겹치면 신청 불가

4) **중복 신청 금지**
- 동일 학생이 동일 강좌(courseId)를 중복 신청 불가

## 3. 불명확한 요구사항에 대한 결정(Stable Decisions)

### 3.1 “이번 학기”의 정의
- 결정: 학기 기간/날짜 범위를 별도로 두지 않고, **서버 실행 시점에 활성화된 단일 학기(Active Term)** 로 간주한다.
- 영향: 모든 수강신청/시간표는 현재 활성 학기 1개(예: termId=1)에 귀속된다.
- 근거: 날짜/타임존 정책이 명시되지 않았고, MVP에서 기간 모델은 과구현이다.

### 3.2 수강신청/취소 가능 기간
- 결정: 서버 실행 중에는 항상 신청/취소 가능(기간 제한 없음).
- 근거: 기간 정책 부재. 기간을 임의로 넣으면 테스트/채점에 변수가 된다.

### 3.3 강좌 학점 범위 및 학점 계산
- 결정: 강좌 credits는 1~3 정수로 생성한다(Seeder).
- 학점 합산: 이번 학기 신청 강좌 credits를 단순 합산한다.
- 근거: “학생 최대 18학점”이 명시되어 있으며, 현실적인 최소 가정.

### 3.4 시간표 모델 및 충돌 정의
- 결정: 강좌 시간은 `요일(dayOfWeek) + 시작(start) + 종료(end)` 로 정의한다.
- 운영 시간 가정: 월~금 09:00~21:00 범위에서 90분 슬롯으로 생성한다.
- 충돌 판정: 같은 요일이고 아래 조건이면 충돌로 판정한다.
    - `start < otherEnd && otherStart < end`
- 휴식시간/간격: 고려하지 않음(요구 없음).
- 근거: “같은 시간대 두 강좌 불가”를 테스트 가능한 규칙으로 단순화.

### 3.5 중복 신청(분반/과목코드)
- 필수: 동일 학생-동일 강좌(courseId) 중복 신청 금지
- 옵션(이번 MVP 미적용): 과목코드(subjectCode) 기반 분반 중복 금지
- 근거: “중복” 범위가 명시되지 않음. 과잉 제약은 채점 리스크.

### 3.6 정원 관리 범위(대기/상태)
- 결정: 대기열 없음. 신청은 `ENROLLED(확정)`만 존재.
- 취소 시 즉시 자리 반환: enrolled 감소
- 근거: 요구 핵심이 “초과 금지/동시성”이므로 상태머신은 과구현.

### 3.7 목록 조회 범위/페이징/정렬
- 결정: 목록 API는 기본 페이징 제공(page, size).
- 기본 정렬: `id ASC`
- 강좌 목록 필수 필드: `id, name, credits, capacity, enrolled, schedule`
- 근거: 학생 10,000 / 강좌 500+이므로 페이징 없이는 비합리적이며, 평가자의 ID 확인을 돕기 위해 정렬을 고정한다.

### 3.8 교수/강좌 관계
- 결정: 강좌당 교수 1명, 교수는 여러 강좌(1:N) 담당 가능
- 근거: 팀티칭 요구 없음. 단순 모델이 테스트/구현 안정적.

### 3.9 “내 시간표” 응답 형식
- 결정: 시간표 항목은 아래 필드를 반환한다.
    - `courseId, courseName, credits, departmentName, professorName, schedule`
- schedule 포맷 예: `"월 09:00-10:30"` (요일 한글 + HH:mm)
- 근거: 강좌 목록 필드와 일관되고, 평가/프론트 확인이 쉽다.

## 4. 에러 응답 표준

### 4.1 공통 에러 스키마
```json
{
  "error": {
    "code": "REG_CAPACITY_FULL",
    "message": "정원이 가득 찼습니다.",
    "detail": {
      "courseId": 1
    }
  }
}
```

## 5. 동시성 제어 전략(핵심)
### 5.1 최종 선택: 강좌별 Lock 기반 임계영역(Per-course Lock)
- 구현: 강좌별로 ReentrantLock을 보유하고, 수강 신청 처리 시 해당 강좌 lock을 획득한다.
- 장점: 구현이 단순하고, 단일 인스턴스 환경에서 동시성 요구를 확실히 만족시킬 수 있다.
- 단점: 멀티 인스턴스 환경에서는 분산 락 또는 DB 락으로 확장 필요.
- 단일 인스턴스 가정: 본 과제 구현은 단일 애플리케이션 인스턴스를 가정한다.
- 멀티 인스턴스 환경에서는 DB 비관적 락 또는 분산 락(예: Redis)으로 확장한다.

### 5.2 원자성(임계영역) 범위
- 수강 신청은 아래 단계를 하나의 임계영역으로 처리한다.
- 정원 확인 → (중복/학점/시간충돌) 검사 → 등록 생성 → enrolled 증가 → 응답
- 이를 통해 동시 요청 상황에서도 enrolled <= capacity를 보장한다.

## 6. 초기 데이터 생성(Seeder) 및 /health

### 6.1 데이터 생성 요구
- 서버 시작 시 동적 생성
- 최소 수량: 학과 10+, 강좌 500+, 학생 10,000+, 교수 100+
- 서버 시작 후 1분 이내 생성 완료
- 생성 중에도 API 호출 가능 (단, 조회 결과는 생성 진행률에 따라 달라질 수 있음)
- 이름 생성: 성씨 30개 + 이름 50개 조합(1,500개) 풀에서 랜덤 배정
- 학생 생성은 병렬 처리(Parallel Stream)로 속도 확보

### 6.2 /health 규격
- GET /health 는 항상 200 OK
- 응답 바디에 seeding 상태와 개수를 포함한다(테스트 편의)

예시:
```json
{
  "status": "UP",
  "seeding": "DONE",
  "counts": {
    "students": 10000,
    "courses": 500,
    "professors": 100
  }
}
```

## 7. 검증 시나리오(테스트 기준)

### 동시성
- Given: capacity=1, enrolled=0 강좌
- When: 100명이 동시에 수강신청 호출
- Then: 성공 1, 실패 99, 최종 enrolled=1

### 학점
- 18학점 채운 뒤 추가 신청 → 409(REG_MAX_CREDITS)
- 시간 충돌
- 겹치는 시간 강좌 신청 → 409(REG_TIME_CONFLICT)

### 중복
- 동일 강좌 재신청 → 409(REG_DUPLICATE_COURSE)

### 취소
- 정상 취소 시 enrolled 감소 및 시간표에서 제거
