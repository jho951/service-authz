# 요구사항 분석 및 결정 사항 (초안)

## 1) 불명확한 부분과 합리적 결정
다음 항목은 문제에서 명확히 정의되지 않아, 질문 없이 합리적으로 결정했다.

1. 학점 제도 및 수강 가능 학점 범위
   결정: 학생별 최대 수강 학점은 18학점, 최소는 0학점. 한 강좌 학점은 1~3 사이 정수로 설정.
2. 시간표/시간충돌 정의
   결정: 강좌는 요일+시작시각+종료시각(30분 단위)로 구성. 시간이 겹치면 충돌로 간주.
3. 중복신청
   결정: 동일 학생-동일 강좌 중복 신청은 금지. 동일 과목 코드의 분반 중복 신청도 금지.
4. 수강/취소 가능 기간
   결정: 서버 실행 중에는 항상 신청/취소 가능(기간 제한 없음).
5. 대기자(대기열) 여부
   결정: 대기자 없음. 정원이 차면 즉시 실패.
6. 교수/학과 관계
   결정: 교수는 정확히 1개 학과에 소속, 강좌는 정확히 1명 교수와 1개 학과에 속함.
7. 에러 코드 정책
   결정: 공통 에러 스키마를 정의하고, HTTP 상태코드 + 앱 에러 코드를 함께 반환.
8. 초기 데이터 생성 방식
   결정: 서버 시작 시 자동 생성(Seeder), 1분 이내. 비동기/배치로 생성하고 완료 전에도 /health는 200.
9. 동시성 제어 범위
   결정: 핵심은 "정원 초과 방지"와 "중복 신청 방지"를 원자적으로 보장.

## 2) 핵심 기능 범위
- 수강 신청
- 수강 취소
- 신청 결과 조회(학생 기준, 강좌 기준)
- 강좌 목록/검색(간단 필터)
- 헬스 체크 (GET /health 항상 200)

## 3) 데이터 초기 생성 요구
- 학과 10+
- 강좌 500+
- 학생 10,000+
- 교수 100+
- 서버 시작 후 1분 이내 생성 완료
- 생성 중에도 API 호출 가능 (단, 조회 결과는 생성 진행률에 따라 달라질 수 있음)

## 4) 도메인 규칙
- 정원(capacity) 초과 불가
- 동일 학생-동일 강좌 중복 신청 불가
- 동일 과목 코드의 분반 중복 신청 불가
- 시간 충돌 시 신청 불가
- 최대 수강 학점(18) 초과 불가

## 5) 에러 코드 정책
공통 응답 형태:

{
"error": {
"code": "REG_CAPACITY_FULL",
"message": "정원이 가득 찼습니다.",
"detail": { ... }
}
}


대표 에러 코드:
- REG_CAPACITY_FULL: 정원 초과
- REG_DUPLICATE_COURSE: 동일 강좌 중복 신청
- REG_DUPLICATE_SUBJECT: 동일 과목코드 분반 중복
- REG_TIME_CONFLICT: 시간 충돌
- REG_MAX_CREDITS: 최대 학점 초과
- REG_NOT_FOUND: 강좌/학생 없음
- REG_ALREADY_CANCELED: 존재하지 않는 수강내역 취소
- REG_SEEDING: 초기 데이터 생성 중, 아직 조회 불가 (필요 시)

## 6) 동시성 제어 전략
### 후보 1: DB 비관적 락 (PESSIMISTIC_WRITE)
- 수강 신청 시 강좌 row를 DB 레벨에서 락
- 정원 확인 + 등록을 트랜잭션으로 처리
- 장점: 단순하고 정확
- 단점: 인메모리 단일 노드 외 환경에서는 DB가 필요

### 후보 2: 인메모리 CAS + 강좌별 락 (선택)
- 강좌별로 `ReentrantLock` 혹은 `Semaphore`를 보유
- 수강 신청 시 강좌 락 획득 → 정원/중복/시간충돌 검사 → 등록
- 장점: 인메모리 환경에서 구현 용이, 즉시성
- 단점: 멀티 인스턴스에서는 분산 락 필요

결정: **인메모리 환경을 우선**하므로 후보 2를 최우선 채택.
향후 DB 기반 동시성 제어로 교체할 수 있도록 서비스 계층에 락 전략을 캡슐화.

## 7) 기술/구현 결정 사항
- 언어: Java
- 프레임워크: Spring Boot
- API: REST
- 저장소: 1단계는 인메모리 저장소(Map 기반), 이후 JPA로 교체 가능
- 시간 충돌 계산은 서버 로컬 기준(타임존 고정)
- GET /health 항상 200
